# Архітектурні правила

Ці правила захищають проєкт від деградації при додаванні функціоналу.
Правила 1–3 перевіряються автоматично у `tests/test_architecture.py` (позначка ✓);
правила 4–6 — конвенції, які тримаються на код-ревʼю (позначка 👁), автоматичних
тестів на них немає. Якщо тест упав — виправляйте код, а не тест (виняток —
свідоме рішення, задокументоване тут). Тести — це guard-rail проти випадкових
порушень, а не «вʼязниця»: навмисно їх обійти можна, але це вже не випадковість.

## Шари

```
handlers  →  services  →  repo  →  models
   ↓             ↓
keyboards    llm (base → gemini)
rendering
```

Залежності йдуть тільки зліва направо. Нижній шар ніколи не знає про верхній.

## Правила

1. ✓ **Моделі (`bot/db/models.py`) — тільки дані.**
   Mapped-колонки + прості read-only `@property` (як `Recipe.category_keys`).
   Жодних методів з логікою, жодних імпортів з services/handlers/aiogram.
   `User` — навмисно «міст» усього коду (identity + `active_group_id` як ключ
   ізоляції груп); він має лишатися худим. Логіка навколо користувача — у repo
   чи services, не в моделі.

2. ✓ **`bot/db/repo.py` — єдина точка SQL.**
   `select`/`delete`/`update` з sqlalchemy — тільки тут. Хендлерам з sqlalchemy
   дозволений лише тип `AsyncSession` для прокидання в repo/services.

3. ✓ **Сервіси (`bot/services/`) не знають про Telegram.**
   Жодного `import aiogram`. Сервіси приймають прості типи (`group_id: int`,
   текст запиту, intent) — не `Message` і не цілий `User`: тенантність
   передається явно через `group_id`.
   **Виняток:** `services/ingestion.py` — свідомий адаптер «Telegram-повідомлення →
   вхід для екстракції», йому aiogram дозволений.

4. 👁 **Хендлери (`bot/handlers/`) — тонкі.**
   Прийняти апдейт → викликати repo/services → відрендерити відповідь
   (`bot/rendering.py`). Без SQL (✓ це перевіряється), без бізнес-правил,
   без промптів. Дані з callback-ів — клієнтські: recipe_id завжди
   перевіряється на належність активній групі (`repo.get_recipe`),
   FSM-контекст (план меню, останній запит) привʼязаний до group_id.

5. 👁 **Нуль галюцинацій (ядро продукту).**
   LLM робить лише дві речі: (а) екстрактує структуру з вхідних даних,
   (б) обирає ID рецептів зі списку кандидатів, який дав код.
   Текст рецепта користувачу — завжди дослівний рендер з БД.
   LLM може формулювати *службові* тексти (уточнювальне питання, коментар
   до меню) — але не зміст рецептів, і кожен такий текст екранується
   (`html.escape`) перед відправкою. Будь-яка нова AI-фіча має проходити
   цей самий фільтр.

6. 👁 **Нова фіча = новий сервіс.**
   Нова логіка йде в окремий файл у `bot/services/`, новий хендлер — тонкий.
   Моделі при цьому отримують щонайбільше нові поля, не поведінку.
   Новий LLM-виклик — через інтерфейс `LLMClient` (`services/llm/base.py`),
   не напряму в SDK провайдера.
